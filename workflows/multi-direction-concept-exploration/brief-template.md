# Brief: <project name>

## User and job
Who the user is and what they need to accomplish, in two or three sentences. Name the
outcome that defines success (for example: a working project with at least one teammate
invited, in one sitting).

## Constraints
- Platform and technical constraints (for example: mobile web required, existing tokens and components only).
- Legal or regulatory constraints (for example: data-residency choice must appear before any content is created).
- Accessibility constraints (for example: WCAG 2.2 AA; no information conveyed by color alone).

## Brand rules
- Voice, density, and visual rules the directions must respect.

## Anti-patterns
- Things the team has already decided against, with the reason (for example: no more than
  one optional step; the current flow has four and completion is 41%).

## Stances (one per direction agent)
Name 3–5 design positions a reasonable designer could defend. Each direction commits
fully to one stance.

- Direction A: <stance, one paragraph>
- Direction B: <stance, one paragraph>
- Direction C: <stance, one paragraph>

## What each direction package must contain
- Concept statement (under 120 words)
- Screen-by-screen structure
- Key interaction decisions
- What this direction deliberately sacrifices
- Effort estimate (S/M/L) with reasoning
- Open questions
